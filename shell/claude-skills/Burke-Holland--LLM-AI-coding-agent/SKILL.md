## 強制性編碼原則

以下編碼原則是強制性的：

1. Structure 結構
- 使用一致且可預測的專案佈局
- 按功能/畫面將程式碼分組；盡量減少共享實用程式
- 創造簡單、清晰的入口點
- 在建立多個文件之前, 首先要確定共享結構。對於跨頁面出現的元素, 請使用框架原生的組合模式 (佈局、基礎範本、提供者、共用元件) 。在多個地方重複編寫相同的程式碼是一種程式碼異味, 而不是應該保留的模式

2. Architecture 架構
- 優先使用扁平化、顯式的程式碼, 而不是抽像或深層結構
- 避免使用巧妙的模式、元編程和不必要的間接調用
- 盡量減少耦合, 以便檔案可以安全地重新生成

3. Functions and Modules
- 保持控制流程線性且簡單
- 使用中小型函數；避免深度嵌套的邏輯
- 明確傳遞狀態；避免使用全域變數

4. 命名和註釋
- 使用 descriptive-but-simple (描述性強但簡潔) 的名稱
- 僅用於記錄不變性、假設或外部需求

5. 日誌和錯誤
- 在關鍵邊界處輸出詳細、結構化的日誌
- 使錯誤訊息清晰明確

6. 可重寫性
- 編寫程式碼時, 應確保任何檔案/模組都可以從頭開始重寫而不會破壞系統
- 優先使用清晰、宣告式的設定 (JSON/YAML/etc.) 

7. 平台使用
- 直接且簡單地使用平台約定 (e.g., WinUI/WPF) , 避免過度抽象

8. 修改
- 擴展/重構時, 遵循現有模式
- 除非另有說明, 否則優先選擇完整檔案重寫而非稍為修改

9. 質量
- 優先考慮確定性、可測試的行為
- 保持測試簡潔, 並專注於驗證可觀察的行為
